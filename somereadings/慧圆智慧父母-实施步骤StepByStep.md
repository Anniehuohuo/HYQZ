# 慧圆智慧父母（微信小程序）Step by Step 实施文档

> 基于当前仓库形态：CRMEB 后端（ThinkPHP）+ uni-app 前端（Vue）+ 后台管理端（template/admin）。
>
> 目标：按“先可跑通、再增强”的方式，最短路径落地：智能体对话 + 推荐卡片（课件）+ 购买与已购交付；矩阵页面走后台动态配置；论坛先做壳子与最小能力。

## Step 0：确定第一阶段范围（当天可完成）

输出物（必须明确到可开发）：
- 4个底部导航固定：首页（对话）、矩阵、论坛、我的
- 1个通用会话页：接收 agentId，完成对话展示与发送输入
- 课件作为“商品”售卖：沿用现有商品/订单/支付能力
- 推荐卡片数据结构：从后端返回（先用假数据也行）

验收标准：
- 前端可以跑通页面跳转与基础UI骨架
- 接口未完成时，页面有 loading/empty/error 三态

## Step 1：先把前端框架搭起来（建议第1周第1-2天）

### 1.1 调整底部导航（TabBar）

现状：tabBar 是“首页/分类/购物车/我的”，配置在 [pages.json](file:///d:/AIProject/MINI/HYQZ/template/uni-app/pages.json)。

动作：
- 将 tabBar 调整为“首页（对话）/智能体矩阵/论坛/我的”
- 对应 pagePath 先指向新建页面路径（页面可以先是空壳）

输出物：
- 新的四个页面路径可正常打开，且不会白屏

### 1.2 新建三页“壳子页面”（可先只做UI骨架）

建议页面（命名可调整）：
- 首页对话：`pages/ai/index`
- 智能体矩阵：`pages/ai/agents`
- 论坛：`pages/forum/index`
- 我的：可沿用现有 `pages/user/index`（先不改结构）

每个页面都先实现：
- 顶部标题/占位内容
- 请求数据的三态：loading/empty/error
- 跳转联调入口（例如：矩阵点一个卡片进入会话页）

输出物：
- UI 走通 + 跳转链路跑通（不依赖真实AI与真实数据）

### 1.3 建立“通用会话页”骨架（主页与矩阵复用）

页面建议：
- `pages/ai/chat`（URL 参数带 agentId）

页面能力（先做最小）：
- 历史消息列表（本地数组即可）
- 输入框 + 发送按钮
- 收到后端响应后追加消息
- 支持渲染“推荐卡片”（先用假数据结构）

输出物：
- 首页进入会话页默认 agentId
- 矩阵页面进入会话页传入 agentId

## Step 2：定义“动态配置”最小闭环（建议第1周第3-5天）

你们的关键诉求是“后台可随时改智谱智能体ID/API Key”等，建议分成两类配置：

### 2.1 展示配置（给小程序用）

包含：
- 智能体列表：名称、分类、icon、简介、排序、启用
- 默认主智能体：defaultAgentId
- 推荐策略开关：是否在首页更积极推荐

输出物：
- 一个移动端接口：`GET /api/agents` 返回列表与默认项

### 2.2 运行配置（给后端调用模型用）

包含：
- provider（智谱）、model、智谱智能体ID（如适用）
- API Key（加密存储）
- prompt、温度、maxTokens 等
- 配置版本号与发布状态

输出物：
- 一个后台管理入口（可用现有“配置/组合数据/自定义表”能力承载，避免从0造后台）
- 一个后端读取配置的服务层方法（按 agentId 获取运行配置）

安全底线：
- 小程序端不出现第三方 API Key
- 后端代理调用模型

## Step 3：打通“聊天接口”（先不流式，降低风险）

### 3.1 先做一次性返回（HTTP）

接口建议：
- `POST /api/ai/chat`
  - 入参：agentId、conversationId（可选）、message
  - 出参：reply、recommendations（推荐卡片数组）、conversationId

输出物：
- 后端可以先“假返回”固定话术 + 固定推荐卡片，保证前后端联调

### 3.2 再接入真实智谱（第二阶段）

顺序建议：
- 先接通单轮对话
- 再加多轮上下文（存 Message/Conversation）
- 最后再做流式输出（如要“打字机效果”，再接 Workerman/WebSocket）

验收标准：
- 小程序端发送一句话能得到回复并渲染
- 推荐卡片在会话流中出现

## Step 4：复用现有“商品/订单/支付”完成购买闭环（并行进行）

现状：项目已有大量商品/订单/支付页面与接口（uni-app 已包含订单、收银台等页面）。

动作：
- 将“课件”作为商品（Product）上架（可先用普通商品类型）
- 推荐卡片里返回商品ID（或跳转链接参数）
- 点击卡片跳转到现有商品详情页/下单页

输出物：
- 购买后在“我的-订单/已购”能看到记录
- 会话页展示“已购”状态（可通过接口查询或在下单回调后刷新）

验收标准：
- 一条推荐卡片可以完成下单支付（测试环境）
- 支付成功后可访问课件内容页（哪怕是简单图文/PDF）

## Step 5：论坛先做“壳子 + 最小UGC”（建议第二阶段）

因为当前仓库更偏商城+CMS文章，论坛UGC大概率需要新增表与接口。

第一版建议只做：
- 列表/详情/发帖/评论/点赞/举报
- 内容安全：走微信内容安全校验 + 举报后台处置

输出物：
- 论坛页从“空壳”升级为可发帖可互动

## Step 6：后台管理端最小改造路径（避免从0开发）

优先策略（从易到难）：
1. 先用现有“系统配置/组合数据/自定义表”承载智能体展示配置与推荐规则
2. 再加“智能体运行配置”的管理入口（含 key 加密、版本、发布、审计）
3. 最后才做更细的运营模块（圈子/帖子管理/推荐位等）

## Step 7：上线前清单（必须项）

- 域名与合法合规：小程序请求域名、业务域名备案、隐私协议与用户授权说明
- 安全：第三方 key 不下发前端；接口鉴权；后台操作审计
- 监控：AI调用失败降级策略（返回兜底话术）；错误日志与告警
- 体验：首屏加载、输入防抖、长文本折叠、网络异常提示

## “第一步先做什么”建议（明确到可执行）

第一步就做 Step 1：
- 先把 tabBar 改为“首页对话/矩阵/论坛/我的”
- 新建三页壳子 + 通用会话页骨架
- 同时定一个最小接口契约：`/api/agents` 与 `/api/ai/chat` 的入参出参

这样做的好处：
- 1-2天内就能看到完整产品形态
- 后端与运营配置可以随后补齐，不会阻塞前端推进
